3계층 프로토콜 IPv4

  • 하는 일

    • 다른 네트워크상에 있는 곳과 데이터를 교환하기 위한 프로토콜이다.
    • 데이터가 정확하게 전달되는 것을 보장하지 않기 때문에 데이터가 손상될 수 있다.(신뢰성 있는 통신은 4계층 프로토콜인 TCP에서 보장해줌)
    • 중복된 패킷을 전달하거나 패킷의 순서를 잘못 전달할 가능성도 있다.

  • 구조

    • 16진수로 표기하고, 20Bytes를 사용한다.
    • Version: 16진수로 항상 4 가 온다. IPv6 의 구조는 아예 다르다.
    • IHL: 16진수로 거의 5 가 온다. 길이는 20~60 사이인데 4로 나눈 값을 적는다.
    • Type of Service: 지금은 사용하지 않는 값으로 00 으로 비워둔다.
    • Total Length: 헤더 + 페이로드를 합한 전체 길이
    • Identification + IP Flags + Fragment Offset
      • Identification: 큰 데이터는 쪼개져서 보내지는데, 받는 쪽에서 원래의 데이터로 합칠 때 동일한 데이터였음을 알아볼 수 있는 id 값
      • IP Flags: 3bit로 첫번째 bit는 항상 0 이다.
        • D(Don’t Fragment): 1 이면 송신측에서 데이터를 안쪼개겠다는 의미. 최대 전송단위보다 큰 데이터는 안쪼개서 보낼 수 없기 때문에 거의 안쓰이는 값이다.
        • M(More Fragment): 1 다른 패킷들이 더 있다는 의미.
      • Fragment Offset: 시작점으로 부터 얼마나 떨어져 있는지의 값을 나타냄. 즉, 받는 쪽에서 쪼개진 데이터를 합칠 때, 얘를 보고 순서대로 조립할 수 있다.
    • Time To Live: 패킷의 유효시간. 라우터 장비를 하나씩 거칠때마다 1씩 줄어든다. 윈도우는 128, 리눅스는 64로 설정한다.
    • Protocol: 상위 프로토콜에 대한 정의
      • ICMP는 01 / TCP는 06 / UDP는 11
    • Header Checksum: 헤더의 오류 유무를 확인한다.
    • Source Address: 4Bytes로 출발지 IP 주소
    • Destination Address: 4Bytes로 목적지 IP 주소
3계층 프로토콜 ICMP(Internet Control Message Protocol)

  • 하는 일

    • 인터넷 상에서 제어 메시지를 주고 받는다.
    • 상대방과 통신 유무를 확인할 때 사용한다.
    • 프로토콜 구조의 Type과 Code를 통해 오류 메시지를 전송받는다.

  • 구조

    • 16진수로 표기하고, 8Byte를 사용한다.

    • Type: 대분류

      • 0 : 통신 유무에 대해 응답한다.

      • 8 : 통신할 수 있는지 확인 요청한다.

      • 3 : 목적지에 도달할 수 없습니다. (주로 경로설정 잘못 했을 때)

      • 11 : 요청시간이 만료되었습니다.(목적지까지 갔는데 응답을 받을 수 없는 경우, 주로 상대방이 방화벽 켜뒀을 때)

    • Code: 소분류

    • Checksum: 헤더의 오류 유무를 확인한다.

라우팅 테이블
  • 목적지까지의 최적의 경로를 지도처럼 저장해둔 테이블
  • $ netstat -r 명령어로 확인할 수 있다.
  • 라우팅 테이블에 등록된 네트워크 대역만 찾아갈 수 있다.
  • 디폴트로 게이트웨이 0.0.0.0 를 설정함으로써 라우팅 테이블에 등록되지 않은 네트워크도 갈 수 있다.
다른 네트워크와 통신 과정
  • 목표: 내 컴퓨터에서 보낸 패킷을 다른 네트워크의 컴퓨터까지 전달한다.

  1. A와 B가 통신하려고 할 때, A의 라우팅 테이블에 B의 네트워크 대역 192.168.20.0이 등록되어 있는지 확인한다. 192.168.20.0 으로 가기 위해 192.168.10.1 로 가라고 설정되어 있다.

  2. ICMP로 통신 확인하기 위해 요청 프로토콜을 작성한다. 요청의 Type은 8 이다.

  3. IPv4 프로토콜 작성한다.

  4. 이더넷 프로토콜을 작성한다. 이때 목적지 MAC 주소는 192.168.10.1 의 게이트웨이의 MAC 주소를 쓴다. 이더넷은 같은 네트워크 대역에서만 사용되니까 여기서 갈 수 있는 데까지의 MAC 주소를 작성한다.

  5. 2계층 장비인 스위치가 이더넷 프로토콜을 확인하고 목적지 MAC 주소까지 패킷을 전달한다.


  6. 공유기가 이더넷 프로토콜 확인해서 나한테 온게 맞으면 IPv4 프로토콜을 확인해본다. 목적지 IP 주소가 다른거니 공유기의 라우팅 테이블을 확인해본다. 192.168.20.0 대역은 192.168.30.2 로 가라고 설정되어 있다.

  7. 공유기가 이더넷 프로토콜을 다시 만들어서 MAC 주소가 22 22 22 22 22 22 인 곳으로 보낸다.


  8. 3계층 장비인 라우터는 이더넷 프로토콜 확인하여 목적지 확인 후 맞으면, IPv4 프로토콜까지 확인해본다. 목적지 IP 주소가 다르니 라우터의 라우팅 테이블을 확인해본다. 192.168.20.0 대역은 192.168.40.2 로 가라고 설정되어 있다.

  9. 라우터가 이너넷 프로토콜을 다시 만들어서 MAC 주소가 44 44 44 44 44 44 인 곳으로 보낸다. 즉, 이더넷 프로토콜은 네트워크 대역이 바뀔 때마다 새로 작성된다.


  10. 위 과정과 마찬가지로 공유기가 이더넷 프로토콜 목적지 MAC 확인후 IPv4 프로토콜 확인한다. 목적지 IP와 다르니 이더넷 프로토콜 새로 작성해서 보낸다.


  11. 2계층 장비인 스위치가 이더넷 프토토콜 확인하고 목적지 MAC 주소로 패킷을 전달한다. 이더넷, IPv4, ICMP 프로토콜 확인해서 요청이 온 것을 확인한다.

  12. B는 ICMP 응답 프로토콜을 작성하여 위와 같은 과ㅡ정으로 A에게 패킷을 전달한다. 응답의 Type은 0 이다.

  13. 응답을 받게 되면 A는 B랑 통신이 된 것을 알 수있다.

통신 과정 정리하면
  1. A →B 가 통신하기 위해 A의 라우팅 테이블 확인 후 ICMP 요청 프로토콜, IPv4 프로토콜 작성, 이더넷 프로토콜 작성
  2. 스위치가 이더넷 확인 후 목적지 MAC 주소로 요청 전달
  3. 공유기가 이더넷 확인 후, 목적지 MAC 주소가 맞으면 IPv4도 확인
  4. 목적지 IP 주소가 다르면 공유기의 라우팅 테이블 확인 후 이더넷 다시 작성
  5. 라우터가 이더넷 확인 후, 목적지 MAC 주소가 맞으면 Ipv4 확인
  6. 목적지 IP 주소가 다르면 라우터의 라우팅 테이블 확인 후 이더넷 다시 작성
  7. 목적지 IP 주소에 도착할 때까지 라우팅 테이블 확인 -> 이더넷 프로토콜 다시 작성해서 전달의 과정을 반복
  8. 목적지 IP 주소가 패킷을 받으면 디캡슐화해서 확인하고, 위의 과정과 동일하게 응답 전달
IPv4의 조각화

  • 정의

    • 큰 IP 패킷들이 적은 MTU(최대 전송 단위, Maximum Transmission Unit, 보통 1500Bytes)를 갖는 링크를 통하여 전송되려면 여러 개의 작은 패킷으로 쪼개어/조각화되어 전송되어야 한다.
    • 목적지까지 패킷을 전달하는 과정에 통과하는 각 라우터마다 전송에 적합한 프레임으로 변환이 필요하다.
    • 일단 조각화되면, 최종 목적지에 도달할 때까지 재조립되지 않는 것이 일반적이다. 최종 수신지에서만 재조립이 가능하다.
    • IPv4에서는 발신지 뿐만 아니라 중간 라우터에서도 IP 조각화가 가능하고, IPv6에서는 IP 단편화가 발신지에서만 가능하다.

  • 구조

    • 보통 First Fragmentation 까지만 조각화를 진행한다.
    • 보낼 데이터의 크기가 11,980 Bytes
    • MTU가 3,300 Bytes인데 IPv4 헤더 20Bytes를 포함해야 하기 때문에 조각화해서 보낼 데이터 크기는 3,280Bytes 이다.
    • MF가 1이면 More Fragment로 뒤에 패킷이 더 있다는 의미이다.
    • Offset는 데이터의 크기만큼 떨어진 값을 의미하는데 8을 나눈 값을 쓴다.(ex. 3280/8=410)
    • First Fragmentation에서 4개로 조각화된 패킷의 Identification은 동일한 값이다.

  • 패킷 조각화하는 과정

    • MTU가 1500 Bytes라 2000Bytes인 데이터를 그냥 보내면 전송되지 않는다. 조각화를 해야한다.
    1. IPv4 헤더 20Bytes를 빼고 1480Bytes, 520Bytes로 데이터를 쪼갠다.
    2. ping 명령은 ICMP 프로토콜을 사용하는데 뒤에 데이터에만 붙는다.
    3. 조각화된 각 패킷에 IPv4 헤더가 붙는다.
    4. 전송되고 이더넷 헤더 14Bytes가 붙는다.
ICMP, IPv4 실습

  • ping 8.8.8.8 요청

    • ICMP 프로토콜의 Type이 8 인 것을 확인할 수 있다.

    • IPv4 프로토콜 구조이다.

  • 8.8.8.8에서 응답

    • ICMP 프로토콜의 Type이 0 인 것을 확인할 수 있다.
