5차 과제의 요구사항을 간략하게 정리해보자

과제의 기한과 요구사항은 다음과 같았다.

  • 기한: 21/11/15 16:00 ~ 11/17 10:00 (약 1.5일)
  • 개발 요구사항:
    • DB: SQLite
    • REST API 기능
      • 임상정보를 수집하는 batch task
      • 수집한 임상정보에 대한 API
      • 수집한 임상정보 리스트 API
        • 최근 일주일내에 업데이트된 임상 정보 리스트
        • pagination 기능
    • ORM 필수 사용
    • 가산점 부분
      • Test 구현
      • 배포하여 웹에서 사용할 수 있도록 제공
      • 임상정보 검색 API
어떻게 과제를 해나갔는가

이번 과제도 팀원 한명이 개인사정으로 참여하지 못하게 되어 4명이 되었고, 첫날은 우리의 루틴대로 프로젝트 이슈 작성, 초기환경 셋팅, 모델링, 마이그레이션까지 진행했다. 보통은 모델링에서 시간이 오래 걸리는데, 이번에는 OpenAPI로부터 데이터를 수집하는 것이어서 쉽게 모델링을 마칠 수 있었다. 구현해야 할 REST API는 총 3개여서 batch task를 두명이 맡기로 하고 나는 상대적으로 난이도가 낮은 임상정보 상세 내용을 보여주는 API를 맡기로 했다.

어려웠던 점은?

난이도가 낮은 API를 구현했기에 코드를 짜는 부분에서 어려운 점은 없었지만, 그 외 부분에서 두가지 어려움이 있었다.

첫번째로 git rebase, squash 전략으로 브랜치를 병합하는 부분이었다.
git rebase하는 부분은 좀 익숙해졌다고 생각했는데, main이 변경되고 충돌이 났을 때 해결해서 다시 rebase하는 과정이 익숙하지 않았다. 충돌난 부분을 수정한 후에는 그 파일을 add만 했어야 했는데, add하고 바로 commit하는게 익숙해 나도 모르게 commit을 해버려서 commit이 너무 많이 생겨 당황스러웠었다ㅋㅋ 이 부분은 팀원의 도움을 받아 해결했고, 전반적인 흐름을 다시 익혔다. 진땀났던 순간이었다ㅜㅜ

두번째로 PR을 날리고 내 코드가 main에 병합되기까지 시간이 오래 걸린 부분이었다.
4번의 과제를 수행하고 나니 우리는 PR에 다른 사람들 코드를 보며 코드리뷰를 남길 수 있는 여유가 생겼다. 나는 코드리뷰를 남길 여유는 안되서 받기만 했지만..ㅎ
url을 앱 이름과 맞추는것, Github에 코드를 올릴 때 마지막에 블랭크 라인을 추가해야 하는 것, django ORM 중 exists() 사용에 대한 리뷰를 받았는데 코드리뷰 자체도 처음이라 코드에 리뷰가 달렸다는 사실만으로 긴장되고 떨렸었다. 사실 이때 exits()를 어떨때 써야하는지 잘 모르고 뭔가 쓰는게 멋있어 보여서 작성했던 부분이었는데 사용 이유에 대해서 설명을 잘 못해서 스스로가 부끄러웠다.

무엇을 느끼고 무엇을 배웠나

내가 짠 코드 한줄 한줄에 대해 명확한 이유를 알아야 불필요한 코드를 작성하지 않게 된다는 것을 정말 마음 속 깊이 느꼈다. 코드 한줄이 성능을 바꿀 수 있어서 더 꼼꼼하게 코드를 작성해야겠다고 생각했다. 그리고 Github에서 파일 끝에 개행 문자를 넣어햐 하는 부분도 새롭게 알게 됬는데 이유는 여기를 참고하면 될 것 같다. 나는 vscode에서 Insert Final Newline이 되게 셋팅해놨다!

6차 과제는 어떻게 하고 싶은가

이제 팀 과제도 얼마 남지 않은것 같은데 남은 과제 동안 지금까지 배웠던 부분을 잘 적용해서 마무리하고 싶다. 어떤 에러도 막을 수 있는 빈틈없는 코드와 단위 테스트, 브랜치 병합, 팀원간의 의사소통까지 더 잘 해서 우수 팀 과제에 뽑히고 싶다.
