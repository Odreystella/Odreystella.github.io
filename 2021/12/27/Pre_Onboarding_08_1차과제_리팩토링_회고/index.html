1차 과제 리팩토링을 진행하게 된 계기

프리온보딩 코스를 이수하고 기능구현도, 배포도 미완이었던 1차 과제에 대해 개인적으로 아쉬움이 많았다. 같이 한 팀원도 코스가 다 끝나면 시간이 좀 여유로울테니 리팩토링해보자고 계속 얘기해왔어서 우리는 하루에 1~2시간 정도 할애하여 1차 과제를 리팩토링하여 완성하기로 했다.

1차 과제의 요구사항을 다시 한번 간략하게 정리해보자

과제의 요구사항은 다음과 같았다.

  • 개발 요구사항:
    • DB: mongodb 사용
    • REST API 기능
      • 원티드 지원 과제 내용 포함(게시판 CRUD API)
      • 게시글 카테고리
      • 게시글 검색
      • 대댓글(1depth)
        • 대댓글 pagination
      • 게시글 읽힘 수
        • 같은 User가 게시글을 읽는 경우 count 수 증가하면 안 됨
    • 유닛테스트 구현
1차과제의 상태와 우리의 목표

1차 과제를 수행하고 한달 이상의 시간이 흘러 지금 과제가 어떤 상태인지 점검했다. 요구사항 중에 게시판 CRUD API와 게시글 카테고리는 작성되어있는 코드로 두기로 했고, 검색 API, 대댓글 API는 리팩토링이 필요했다. 게시글 읽힘 수 API는 새로 코드를 짜야했다. 그리고 mongodb와 연동하면서 유닛테스트를 돌렸을 때, 해결하지 못했던 에러를 해결해서 테스트를 성공시켜야 했고, swagger를 처음 적용하여 Query String을 전달하지 못했던 부분을 성공시켜야 했다. 모든 기능구현과 테스트가 끝난 후에 배포와 README.md를 다시 정리하기로 했고, 더 나아가 우리는 도커로 배포를 해보기로 했다. 정리하자면 검색 API, 대댓글 API 리팩토링, 게시글 읽힘 수 API 구현, 유닛테스트 구현, 도커를 사용하여 배포하는 것이 우리의 목표였다.

어떻게 과제를 해나갔는가

우리는 문제를 해결해나가면서 공부하는 것이 목적이었기 때문에 각자 업무를 나누는 것보다 페어프로그래밍을 진행하는 것이 의미있을 것이라고 판단했다. 늘 하던대로 깃허브에 이슈를 작성했고, 온라인에서 만나 가능한 하루에 하나씩 해결해 나가기로 했다. 5개의 이슈를 작성했고, 총 일주일정도 걸렸던것 같다.

어려웠던 점은?

인원이 많든 적든 협업할 때 의견을 조율하는 일은 늘 어렵다. 내가 경험한 만큼 이해할 수 있다고 생각하는데, 나와 다른 경험을 가지고 있는 상대방의 의견을 이해할 때는 많은 대화가 필요한 것 같다. 모델에서 사용하지 않는 필드를 지우는 일, 에러 메시지를 작성하는 일, 변수의 이름을 정하는 일, 에러 핸들링의 범위를 정하는 일 등 어떻게 보면 사소할 수 있지만 정답이 없기 때문에 충분히 의견차이가 있을 수 있는 일에서 대화를 통해 서로 설득해가며 의견을 조율했다.

무엇을 느끼고 무엇을 배웠나

도커로 배포에 성공했을 때 일단 뿌듯했다. 처음 과제가 주어졌던 약 한달 전보다 코드가 눈에 더 잘 들어온다고 느꼈고, 에러가 발생해도 구글링해서 전보다 더 잘 해결할 수 있었다. 한달하고 조금 넘는 시간동안 그래도 성장했다는 것이 느껴져서 기뻤고, 자신감도 생겼다.

그리고 어떤 경험이든 항상 배울점은 있다. 이번 리팩토링을 통해 지난번에는 못했던 swagger로 Query String을 전달하는 방법을 찾았다. query_serializer와 manual_parameters라는 인자를 사용해서 전달할 수 있었는데 후자를 선택하여 검색 API와 대댓글 조회 API를 잘 구현할 수 있었다. 또한, Query String과 Path Variable의 차이를 공부했고, 도커의 명령어와 실행 흐름에 더욱 익숙해질 수 있었다.
